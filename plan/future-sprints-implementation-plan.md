# historia Sprint 5以降 実装計画

## 目的
- `Sprint 4` までで完成した MVP を前提に、次段階の拡張を無理なく進める。
- 追加機能を「可視化」「データ運用」「高度モデル化」に分け、依存関係を崩さず段階的に実装する。
- 実装対象ごとに、着手条件、スコープ、完了条件、先送り条件を明確にする。

## 前提
- `Sprint 1-4` は完了済み。
- 現在の中核は `Event` と各主体の CRUD、相互リンク、戦争・乱、一覧検索、seed データ。
- 次段階では、既存 CRUD を壊さずに「見せ方」「データ移送」「モデル深掘り」を追加する。

## 全体方針
- `Sprint 5`: まずは既存データを活かす可視化と探索性の強化を行う。
- `Sprint 6`: 外部入出力、変更履歴、出典など、データ運用機能を足す。
- `Sprint 7`: 国家変遷や継承関係など、既存モデルを拡張する高度構造に着手する。
- 各スプリントで DB 変更を含む場合は、必ず `seed` と後方互換の migration をセットで更新する。

## Sprint 5: 可視化と探索強化
### 目的
- 登録済みデータを「辿れる」だけでなく「俯瞰できる」状態にする。
- 詳細ページ中心の探索から、グラフとタイムライン中心の探索へ広げる。

### 対象機能
#### 1. イベント関係グラフビュー
- `/graph/events` を追加する。
- 初期対象ノードは `Event` のみとし、`EventRelation` をエッジとして可視化する。
- 表示対象の関係種別は `before / after / cause / related` を切り替えられるようにする。
- 検索条件として `eventType`、期間、関連主体、タグを受け取れるようにする。
- ノードクリックでイベント詳細へ遷移できるようにする。

#### 2. 横断グラフの第一段階
- イベントに加えて `Person`、`Polity`、`Religion`、`Region` を補助ノードとして表示できるようにする。
- 初期版では「イベントと直接リンクされた主体のみ」を表示対象にし、主体間専用エッジは持たない。
- グラフ密度が高くなりすぎる場合に備えて、`イベントのみ / イベント+人物 / イベント+国家...` の表示切替を用意する。

#### 3. 時系列タイムライン
- `/timeline` を追加する。
- 初期対象は `Event`、`HistoricalPeriod`、`Polity`、`Dynasty`、`Religion`、`Sect`。
- 年代範囲を横軸にし、単年イベントと期間主体を同じビューに載せる。
- `BCE/CE`、範囲年、不明年、継続中を破綻なく表示できることを完了条件にする。
- 初期版では年単位固定。月日 UI は含めない。

#### 4. 一覧検索の第二段階
- 保存済みフィルタ条件を URL だけでなく UI 上の保存セットとして扱えるようにする。
- 並び順を追加する。最低限 `年代順`、`名前順`、`更新日順`。
- 一覧から複数件を選択し、比較または一括遷移できる導線を追加する。

### 技術方針
- 可視化ライブラリは `React Flow` または同等の軽量ライブラリを候補にする。
- タイムラインは既成ライブラリで無理に解決せず、初期版は独自のシンプル実装でもよい。
- 可視化データは `server/services` で専用 DTO を返し、ページ側で無加工に近い形で描画できるようにする。

### データ/API計画
- `getEventGraphView(filters)` を追加する。
- `getTimelineView(filters)` を追加する。
- グラフ・タイムライン用 DTO は `src/types/visualization.ts` へ切り出す。
- ノードラベル、表示期間、関係種別、遷移先 URL を DTO に含める。

### 完了条件
- イベント関係グラフビューが動作し、主要イベントの因果・前後関係を視覚的に辿れる。
- タイムラインで単年イベントと期間主体を同時に見られる。
- 一覧検索に並び順と保存済みフィルタ相当の再利用導線がある。

### 先送り条件
- 全主体を同時に載せた密な巨大グラフ。
- 自動クラスタリングや高度なレイアウト最適化。
- 月日粒度のタイムライン。

## Sprint 6: データ運用
### 目的
- ローカル運用を前提とした、投入・修正・追跡の機能を整える。
- 手入力中心の運用から、外部データ流入と保守に耐える形へ進める。

### 対象機能
#### 1. import/export
- JSON export を最優先で実装する。
- 次に JSON import を実装する。初期版は「全置換」ではなく「追加入力 + ID/名前ベースの重複検知」を基本にする。
- CSV は `Event` と `Person` のみ対象にした限定版から始める。
- エクスポート対象には `Event`、各主体、リンク、中間テーブル、関係データを含める。

#### 2. 出典・参考文献管理
- `Source` と `Citation` を追加する。
- `Source` は書誌情報または URL を持てるようにする。
- `Citation` は `Event`、`Person`、`Polity`、`HistoricalPeriod`、`Religion` などへ紐づけられるようにする。
- 初期版は引用箇所の自由記述と簡易メモまででよい。

#### 3. 変更履歴
- 監査ログの完全版ではなく、まずは変更履歴の簡易版を入れる。
- 更新前スナップショットを `history_*` テーブルか汎用履歴テーブルへ保存する。
- 少なくとも `Event`、`Person`、`Polity`、`HistoricalPeriod` を対象にする。
- 詳細画面から「直近の変更」を確認できるようにする。

#### 4. 重複検知
- import 時に、`名前一致`、`別名一致`、`年代近接` を使った候補表示を行う。
- 自動マージはしない。候補提示と手動選択に留める。

### 技術方針
- import/export は `server/services/import-export.ts` を新設して集約する。
- JSON スキーマを明示し、バージョン番号を持たせる。
- 出典と履歴は既存 CRUD を壊さないように、付加テーブルとして追加する。

### データ/API計画
- `Source`
  - `id`
  - `title`
  - `author`
  - `publisher`
  - `publishedAtLabel`
  - `url`
  - `note`
- `Citation`
  - `id`
  - `sourceId`
  - `targetType`
  - `targetId`
  - `locator`
  - `quote`
  - `note`
- `ChangeHistory`
  - `id`
  - `targetType`
  - `targetId`
  - `snapshotJson`
  - `changedAt`

### 完了条件
- JSON export/import でローカルバックアップと再投入ができる。
- 主要主体に出典を紐づけて表示できる。
- 少なくとも主要 4 モデルで変更履歴を確認できる。

### 先送り条件
- Google Sheets や外部 API との同期。
- 自動マージ。
- 権限付き監査ログ。

## Sprint 7: 高度モデル化
### 目的
- MVP では単純リンクに留めていた歴史構造を、専用モデルとして扱えるようにする。
- 長期的に「国家変遷」「王朝継承」「宗派分派」「地域関係」を表現できる土台を作る。

### 対象機能
#### 1. 国家変遷
- `PolityTransition` を追加する。
- `predecessorPolityId`、`successorPolityId`、`transitionType`、`timeExpression`、`note` を持つ。
- `split`、`merge`、`rename`、`succession` などを扱えるようにする。

#### 2. 王朝継承
- `DynastySuccession` を追加する。
- 同一国家内の王朝交代を表現できるようにする。
- `Polity` 詳細で継承順を表示できるようにする。

#### 3. 宗派分派
- `Sect` の親子関係を 1 段ではなく多段階に拡張する。
- 既存の `religionId` 必須は維持しつつ、`parentSectId` を追加する。
- 宗派ツリーを詳細画面と簡易グラフで表示できるようにする。

#### 4. 地域関係
- `RegionRelation` を追加する。
- `contains`、`adjacent`、`cultural_sphere` を最低限扱う。
- 地域の階層ツリーとは別に、文化圏や隣接関係を扱えるようにする。

#### 5. 時代区分の包含・継承
- `HistoricalPeriodRelation` を追加する。
- `contains`、`overlaps`、`succeeds` を扱う。
- 同一カテゴリ内、またはカテゴリ横断での時代区分比較を可能にする。

### 技術方針
- 既存リンクだけで無理に表現していた関係を、専用テーブルへ段階的に分離する。
- 各専用関係モデルは、詳細画面で表示してから一覧や可視化へ広げる。
- migration では既存データを壊さず、空テーブル追加から始める。

### データ/API計画
- `PolityTransition`
- `DynastySuccession`
- `RegionRelation`
- `HistoricalPeriodRelation`
- `Sect.parentSectId`

### 完了条件
- 国家変遷、王朝継承、宗派分派、地域関係、時代区分関係を UI から登録・表示できる。
- 少なくとも詳細画面で専用関係を辿れる。
- 既存の CRUD や seed が migration 後も破綻しない。

### 先送り条件
- 地図表示や GIS 連携。
- 自動推論による変遷候補生成。
- 関係の整合性自動検証。

## 推奨実装順
1. Sprint 5 のイベントグラフ。
2. Sprint 5 のタイムライン。
3. Sprint 6 の JSON export。
4. Sprint 6 の JSON import。
5. Sprint 6 の出典。
6. Sprint 6 の変更履歴。
7. Sprint 7 の国家変遷。
8. Sprint 7 の王朝継承。
9. Sprint 7 の宗派分派。
10. Sprint 7 の地域関係と時代区分関係。

## リスクと対策
- 可視化が重くなる:
  - 対策: 初期表示件数制限、関係種別フィルタ、ノード種別切替を入れる。
- import が既存データを壊す:
  - 対策: 初期版は dry-run と preview を必須にする。
- 高度モデル化で既存 UI が複雑化する:
  - 対策: 新関係はまず詳細ページの別セクションに追加し、既存フォームを急に巨大化させない。

## この計画の使い方
- `Sprint 5` 着手時は、このファイルを元に `tasks.md` の後続版を別ファイルで切る。
- DB 変更を伴う項目は、必ず `schema`、`migration`、`seed`、`テスト` を 1 セットで扱う。
- 各スプリント終了時に、このファイルへ `実装状況` を追記して更新する。
