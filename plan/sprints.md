# historia MVPスプリント計画

## 前提
- 1 スプリントは 1 週間前後を想定する。
- 1 人開発でも回せる粒度にしつつ、複数人でも分担しやすいようにまとめる。
- 依存関係を崩さないことを優先し、後工程の手戻りが大きい項目は前倒しする。
- タスク番号は [tasks.md](/Users/sky0621/work/github.com/sky0621/historia/plan/tasks.md) に対応する。

## Sprint 1: 基盤固定
### 目的
- アプリが起動し、DB と UI の最低限の土台ができている状態にする。
- 後続の全画面で使う共通型と共通 UI を先に固定する。

### 対象タスク
- T001 プロジェクト初期化
- T002 共通設計値の定義
- T003 DBスキーマ初版
- T004 DBアクセス層
- T005 共通UI骨格

### 成果物
- Next.js + SQLite + ORM の実行可能な開発環境
- 初回マイグレーション
- 共通 `TimeExpression`
- 共通 enum / 型
- 共通レイアウトとフォーム骨格

### 完了条件
- ローカル起動できる。
- DB マイグレーションが通る。
- `TimeExpression` を UI から入力できる。
- CRUD 実装に再利用できるアクセス層が存在する。

### リスク
- `TimeExpression` の設計が曖昧なままだと全後続タスクに波及する。
- ORM 選定やスキーマ設計がぶれると Sprint 2 以降の UI 実装が止まる。

## Sprint 2: 主要マスタ整備
### 目的
- 歴史データの土台となる主体を一通り登録できる状態にする。
- Event 実装前に参照先マスタを揃える。

### 対象タスク
- T006 Region CRUD
- T007 PeriodCategory CRUD
- T008 Polity / Dynasty CRUD
- T009 Religion / Sect CRUD
- T010 Person CRUD と役職履歴
- T011 HistoricalPeriod CRUD

### 成果物
- 地域、国家、王朝、宗教、宗派、人物、時代区分、カテゴリの CRUD 画面
- 人物の役職履歴管理
- 地域階層と各主体の地域リンク

### 完了条件
- 主要主体を UI から登録・編集・削除できる。
- 人物に役職履歴を登録できる。
- 宗教/宗派、国家/王朝、カテゴリ/時代区分の親子関係が保存できる。
- 各主体に複数地域を紐づけられる。

### 並行化の目安
- 1人なら `Region -> Polity/Dynasty -> Religion/Sect -> Person -> HistoricalPeriod` の順。
- 複数人なら `Region/Category`、`国家/王朝`、`宗教/宗派`、`人物/時代区分` に分けやすい。

## Sprint 3: イベント中核
### 目的
- 年表システムとしての中核であるイベント登録と、相互リンクの基本導線を成立させる。

### 対象タスク
- T012 Event CRUD 基本
- T013 EventRelation
- T018 詳細画面の相互リンク

### 成果物
- 通常イベントの一覧・詳細・編集画面
- イベントと各主体の関連リンク
- イベント間関係
- 詳細画面から詳細画面への相互遷移

### 完了条件
- 通常イベントを登録できる。
- 人物、国家、王朝、時代区分、宗教、宗派、地域をイベントに紐づけられる。
- `before/after`、`cause/related` を登録できる。
- 主要詳細画面の関連リンクを辿れる。

### リスク
- 多対多保存処理が複数画面でばらつくと不整合が起きやすい。
- 詳細導線を後回しにしすぎると Sprint 4 の確認効率が落ちる。

## Sprint 4: 戦争・検索・仕上げ
### 目的
- 戦争・乱の専用要件を載せ、MVP として使える検索性と検証環境を整える。

### 対象タスク
- T014 戦争・乱フォーム分岐
- T015 ConflictParticipant
- T016 ConflictOutcome
- T017 一覧検索
- T019 Seedデータ
- T020 動作確認

### 成果物
- 戦争・乱イベントの専用入力 UI
- 参加者と結果の管理
- 主要一覧の検索機能
- 初期データ
- MVP 動作確認結果

### 完了条件
- 戦争・乱を期間付きイベントとして登録できる。
- 参加者と結果を構造化して保存できる。
- 主要一覧で最低限の複合検索が使える。
- Seed データで主要導線を再現できる。
- MVP 受け入れ条件を満たす。

### 仕上げ判断
- Sprint 4 終了時点で、全体グラフビューや import/export には着手しない。
- 不足が出た場合も、まずは CRUD・検索・相互リンクの安定を優先する。

## Sprintごとのマイルストーン
- Sprint 1 完了時: 実装開始可能な基盤がある。
- Sprint 2 完了時: 主要マスタを登録できる。
- Sprint 3 完了時: 年表としてイベント運用できる。
- Sprint 4 完了時: 戦争・乱を含む MVP が一通り使える。

- Sprint 6 候補: import/export、出典管理、変更履歴
- Sprint 7 候補: 国家変遷、王朝継承、宗派分派、地域関係モデル
