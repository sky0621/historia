# historia MVP実装計画

## 目的
最新版要件をそのまま一度に実装すると、モデル数と関連が多く初期立ち上げコストが高い。そこで、初回リリースを `MVP` と `将来拡張` に分離し、まずは「歴史データを破綻なく登録・検索できる」状態を最短で作る。

この計画では、MVP を実装者がそのまま着手できる粒度に分解し、後続で追加すべき拡張対象も固定する。

## MVP方針
- 技術基盤は `Next.js App Router + TypeScript + SQLite + ORM` を採用する。
- まずはデータモデルと CRUD を安定させ、可視化や高度な関係表現は後ろに送る。
- 共通 `TimeExpression` を最優先で整備し、曖昧年・BCE/CE・範囲年を全主要モデルで一貫して扱う。
- 主体間の専用関係モデルは最小限に絞り、MVP では「リンクできる」「検索できる」「詳細画面で辿れる」を成立条件にする。

## MVPスコープ
### 1. 基盤
- Next.js プロジェクト初期化。
- SQLite 接続、ORM 導入、初回マイグレーション整備。
- レイアウト、ナビゲーション、一覧/詳細/編集の共通 UI 骨格を用意する。
- 共通フォーム部品、共通検索 UI、共通バリデーション方針を定義する。

### 2. 共通モデル
- `TimeExpression` を共通値オブジェクトとして実装する。
- 最低限の主モデルとして `Event`、`Person`、`Polity`、`HistoricalPeriod`、`Region`、`Religion`、`Sect`、`PeriodCategory` を実装する。
- `Dynasty` は MVP に含めるが、国家配下の単純な CRUD と関連表示までに留める。
- `RoleAssignment` を実装し、人物に役職履歴を持たせる。
- 各主体と `Region` の多対多リンクを実装する。

### 3. Event中心機能
- `Event` の一覧・詳細・作成編集を実装する。
- `eventType` は MVP で `general`、`war`、`rebellion`、`civil_war` を持つ。
- イベントに人物、国家、王朝、時代区分、宗教、宗派、地域を紐づけられるようにする。
- 一覧画面で年代、イベント種別、関連人物、関連国家、関連地域、関連時代区分による絞り込みを提供する。
- `EventRelation` は MVP に含め、`before/after` と `cause/related` を登録・表示できるようにする。

### 4. 主体管理
- `Person` の一覧・詳細・作成編集を実装する。
- 生年、没年、別名、役職履歴、関連地域、関連宗教/宗派、関連時代区分を扱えるようにする。
- `Polity`、`Dynasty` の一覧・詳細・作成編集を実装する。
- `HistoricalPeriod` と `PeriodCategory` の一覧・詳細・作成編集を実装する。
- `Religion` と `Sect` の一覧・詳細・作成編集を実装する。宗派は必ず宗教に属させる。
- `Region` の一覧・詳細・作成編集を実装し、親地域による階層管理を可能にする。

### 5. 戦争・乱のMVP
- 戦争・乱は `Event` の種別として実装する。
- `startTimeExpression` と `endTimeExpression` を入力できるようにする。
- `ConflictParticipant` を実装し、国家、人物、宗教、宗派を役割付きで複数登録できるようにする。
- 戦争・乱の結果は MVP では `ConflictOutcome` の簡易版として実装し、勝者側、敗者側、講和要約、メモを保存できるようにする。
- 条約や講和は別イベントとして登録し、関連イベントリンクで接続できるようにする。

### 6. MVPで含める画面
- ダッシュボードは不要。トップはイベント一覧でよい。
- 画面は `イベント`、`人物`、`国家`、`王朝`、`時代区分`、`宗教`、`宗派`、`地域` の一覧・詳細・編集を用意する。
- 関係可視化は MVP では「簡易表示」に留め、詳細画面で関連リンク一覧を確認できればよい。

## MVPで後回しにするもの
- 全体グラフビュー。
- 高度な戦争結果モデル。
- 地域間の専用関係モデル。
- 宗派の多段階分派ツリー。
- 国家・王朝の継承関係や領域変遷の厳密モデル。
- 時代区分同士の包含・継承関係。
- import/export。
- 認証、権限、共有、監査ログ。
- 月日粒度の UI。

## 実装順
### フェーズ1: プロジェクト基盤
- Next.js、ORM、SQLite、マイグレーション基盤を整える。
- 共通 `TimeExpression` と主要 enum を定義する。
- 共通レイアウトと CRUD の最小 UI 骨格を作る。

### フェーズ2: 主体モデル
- `Region`、`PeriodCategory`、`Polity`、`Dynasty`、`Religion`、`Sect`、`Person`、`HistoricalPeriod` を実装する。
- 各モデルの CRUD と一覧検索を用意する。

### フェーズ3: Event
- `Event`、`EventRelation`、各主体とのリンクを実装する。
- イベント一覧・詳細・編集と複合検索を実装する。

### フェーズ4: 戦争・乱
- `eventType` に応じたフォーム切替を実装する。
- `ConflictParticipant` と簡易 `ConflictOutcome` を実装する。

### フェーズ5: 仕上げ
- 詳細画面間の相互リンクを整える。
- 主要一覧の検索導線を整える。
- 最低限の seed データを用意し、動作確認を行う。

## MVP受け入れ条件
- イベント、人物、国家、王朝、時代区分、カテゴリ、宗教、宗派、地域をローカルで CRUD できる。
- BCE/CE、推定年、範囲年、不明年を主要モデルで保存・表示できる。
- 人物に役職履歴を登録でき、国家または王朝を紐づけられる。
- 宗教・宗派に開始終了年と複数開祖を登録できる。
- 地域を階層管理でき、各主体に複数地域を紐づけられる。
- 戦争・乱を期間付きイベントとして登録でき、参加者と結果を保存できる。
- イベント一覧で複合検索でき、関連主体の詳細へ遷移できる。

## 将来拡張
### 可視化
- イベント関係の全体グラフビュー。
- 人物・国家・宗教・地域を含む横断グラフ。
- 時系列タイムライン表示。

### モデル拡張
- 王朝継承、国家変遷、領域変遷のモデル化。
- 宗派の多段階階層化。
- 時代区分同士の包含・継承関係。
- 地域間の隣接・包含・文化圏関係。
- 戦争の参加勢力として王朝・地域・任意勢力を追加。

### データ運用
- JSON/YAML/CSV import/export。
- 一括編集。
- データ重複検知。
- 出典・参考文献管理。

### プロダクト機能
- 認証、ユーザー管理、共有。
- 監査ログ、変更履歴。
- お気に入り、ブックマーク、比較表示。

## 実装メモ
- `MVP` の成功条件は「広く浅く全部載せ」ではなく、「主要主体を一通り登録・相互参照できること」に置く。
- 戦争・乱、王朝、宗教、時代区分は MVP に含めるが、深い構造化は避ける。
- 迷ったら「構造は将来拡張可能に保ち、UI と初期制約は単純にする」を優先する。
